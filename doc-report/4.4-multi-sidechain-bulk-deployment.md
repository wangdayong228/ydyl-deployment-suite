## 4.4 多类型侧链规模化部署

### 4.4.1 问题分析与研究目标

在异构 L2 栈共存的环境下，如何以可恢复、可观测、对后续跨链压测友好的方式，将 96 条 XJST、2 条 CDK（zk-rollup）、2 条 OP Stack（op-rollup）共 100 条侧链一次性部署到 AWS EC2 上，并保证产出的元数据足以驱动后续的跨链消息验证与 TPS 测试？

这一问题的挑战并非来自单一缺口，而是分布在异构栈差异、状态化部署长尾、网络拓扑约束和元数据契约四个维度。四类挑战各自独立——解决任意一类不能消除其余三类。

**C1 异构栈差异。** CDK 走 Kurtosis 编排的多容器栈（cdk-erigon / cdk-node / zkevm-prover），跨链中继依赖 ZK proof；OP Stack 同样走 Kurtosis 但容器组合不同（op-node / op-geth / op-batcher / op-proposer），跨链中继走 Merkle proof；XJST 走自研脚本部署 conflux-rust 的 xjst 模式节点，无独立中继服务，但需要在链内多节点间协调 L1 合约部署与地址分发。三套形态各自有独立的流水线脚本（`cdk_pipe.sh` / `op_pipe.sh` / `xjst_pipe.sh`）、独立的环境变量契约和独立的产物落点，不可能用单一脚本统一编排。

**C2 多步状态化部署。** 每条链 12 步流水线覆盖从密钥生成到 PM2 巡检的完整链路。单条 CDK 链的部署含 Kurtosis 多容器启动、L1 合约部署、claim-service 拉起等长尾步骤，单步失败不能从零重启——流水线必须支持断点续跑，且续跑时已完成的步骤不能重复执行，否则可能覆盖已有合约部署产物或重置 claim-service 状态。

**C3 拓扑约束。** 节点到 L1 RPC 的网络延迟一旦升高，OP/CDK 的 L2 derivation 拉取 L1 数据的速度会持续慢于 L1 出块速度，L2 所引用的 L1 head 与真实 L1 head 之间的滞后量随时间单调增长。这一滞后是累积型而非瞬时型——跨 region/AZ 拓扑下尤为明显，将"批量部署"从纯软件问题升级为部署拓扑约束。需要说明的是，4.2 节 C3 描述的是触达 `maxSequencerDrift` 硬阈值（1800 秒）后的 sequencer 瞬时停摆，而本节描述的是更早期、更普遍的 L2 head 持续滞后现象——两者是相邻而非同一问题。

**C4 元数据契约。** 100 条链各自产出 RPC 端点、bridge 合约地址、Counter 合约地址等元数据。若无统一格式与统一获取入口，下游的 `gen-cross-tx-config` 无法自动扫描所有链并生成跨链压测 jobs，部署产物与压测链路之间的人力手工衔接将成为规模化瓶颈。元数据契约的定义不仅关乎格式一致性，还要求每条链的 `ydyl-console-service` 健康运行以提供实时查询能力。

对应这四类挑战，本节的研究目标以四条量化指标定义：

1. 单条命令一次性发起 ≥100 条侧链批量部署；失败可定向重试不重建已成功节点
2. 部署完成后无人工整理即可生成 8 路并行跨链压测 jobs
3. 100 条链上累计创建 ≥10 亿用户账户——这一规模是侧链可承载真实业务负载的硬指标，由 `ydyl-gen-accounts` 在每条链的 step9 批量生成与充值
4. 96 XJST + 2 CDK + 2 OP，跨链 Counter 调用全链路可达

### 4.4.2 总体部署架构

#### 4.4.2.1 控制平面与数据平面分离

100 条异构侧链的批量部署首先要求一个清晰的架构边界：哪些职责由集中式编排承担，哪些职责下放给各节点自治。本方案将系统划分为控制平面与数据平面，两者之间的边界由一条原则定义——控制平面不持有任何链状态。

**控制平面无状态。** `ydyl-deploy-client`（Go/Cobra CLI）是控制平面的唯一组件，负责按 `config.deploy.yaml` 中 `services[]` 的声明批量创建 EC2 实例并远程派发对应的流水线脚本。它不持有任何链的私钥、合约地址、RPC 端点或状态文件——这些全部落在数据平面各节点的本地文件系统中（`*-work/output/`、`*_pipe.state`）。控制平面失联后已部署节点继续独立运行，重启控制平面后通过读取各节点的 `servers.json` 与 `script_status.json` 即可恢复对全局部署状态的感知，无需访问链上状态。

**数据平面节点的标准形态。** 每台 EC2 实例在部署完成后承担三类角色。第一，L2 节点本体：CDK 是 Kurtosis 编排的多容器栈（cdk-erigon / cdk-node / zkevm-prover），OP 是 op-node / op-geth / op-batcher / op-proposer 四进程，XJST 是 conflux-rust 的 xjst 模式单进程。第二，跨链中继服务：CDK 配 `zk-claim-service`（监听 L2 出口事件、拉取 ZK proof 并提交 L1），OP 配 `op-claim-service`（Merkle proof 生成与 L1 中继），XJST 无独立中继服务——其跨链对账由 bridge 合约直接完成。第三，元数据服务 `ydyl-console-service`（Go/Gin HTTP API），对外暴露该链的 RPC 端点、合约地址、部署状态等结构化元数据。

`jsonrpc-proxy` 不在节点标准形态中，其部署拓扑因栈而异。OP 栈下所有 OP 链共享同一个 `jsonrpc-proxy` 实例——共享的根因来自 4.2 节 C1：proxy 对 OP 节点做 block hash 修正时需要从同一起始 block 进入 derivation，才能保证修正后的 hash 在所有 OP 节点间一致，这一有状态特性要求单实例。CDK 栈下每条链独立一个 `jsonrpc-proxy`：CDK 不在代理层做 hash 修正（其 hash 校验冲突由内核侧解决，详见 4.2 节），代理仅承担 RPC 转发与 gas price 补偿等无状态职责，可一链一实例且无跨链协同需求。XJST 不需要 `jsonrpc-proxy`——其 RPC 直连 L1 节点，无 hash 适配需求。OP 共享与 CDK 独立的根因将在 4.4.3.2 完整展开。

**节点间关系。** 96 条 XJST 链的每条链内部存在多节点协同——同一条 XJST 链的 node-2..node-N 需从 node-1 拉取该链专属的 L1 合约地址组（`simple_calculator` / `state_sender` / `unified_bridge`），但跨链之间各 XJST 完全独立。CDK 与 OP 链彼此独立，无链间协同。所有 100 条链共享单一 `L1_BRIDGE_HUB_CONTRACT`——这是把异构侧链编织成统一互操作网络的唯一全局单点。

#### 4.4.2.2 网络拓扑设计

所有 100 条侧链的 EC2 实例部署在同一 AWS region 的同一可用区（AZ）内。这一决策的首要目的不是简化运维，而是压低节点到 L1 RPC 的网络延迟。

RPC 延迟与 L2 head 滞后之间的关系是累积型的：当 L1 RPC 查询延迟升高时，OP/CDK 的 derivation 管线每次拉取 L1 区块数据的耗时增加，单次拉取延迟积累到一定程度后，L2 对 L1 区块的消费速率持续低于 L1 出块速率，L2 所引用的 L1 head 与真实的 L1 head 之间的时间差随运行时间单调增长。跨 region 部署时 RPC 往返延迟可达数十至数百毫秒，L1 每秒产出一个区块，延迟的持续存在使 L2 head 每一秒都在落后更多——这不是一个突发故障，而是一个静默累积的劣化过程。

与 4.2 节 C3 的关系需要明确界定。4.2 C3 描述的是当上述滞后量触达 OP Stack 的 `maxSequencerDrift` 硬阈值（Fjord 分叉后为 1800 秒）时，sequencer 拒绝产出新 L2 区块——这是一个瞬时停摆事故。本节描述的是更早期、更普遍的累积滞后现象：在触发任何硬阈值之前，L2 head 的持续落后已经使跨链消息的时效性劣化，且这种劣化在跨 region/AZ 拓扑下尤为明显。两者是相邻而非同一问题——4.2 C3 是滞后累积到极值后的终态，本节是滞后累积的过程本身。

**L1 共享面边界。** `L1_BRIDGE_HUB_CONTRACT` 是 100 条链唯一全局共享的 L1 合约。与之并列的另一组 L1 合约——XJST 的 `unified_bridge` / `state_sender` / `simple_calculator`、CDK 的 rollup 合约组、OP 的 `OptimismPortal2` 等——均按链独立部署，不存在跨链共享。L1 合约的共享与独立边界精确落在 `BRIDGE_HUB`（全局注册表）与各栈本链桥实现（按链独立）之间。

#### 4.4.2.3 公共依赖与组件清单

三类侧链共享一套公共组件集，差异仅在各组件在不同栈中的实例数与部署拓扑。下表按组件角色分组，横向标注三栈的差异，作为 4.4.3 节详解每条流水线的总览索引。

| 组件 | 角色 | CDK | OP | XJST |
|------|------|-----|-----|------|
| `*_pipe.sh` | 编排入口 | cdk_pipe.sh | op_pipe.sh | xjst_pipe.sh |
| L2 节点 | 链本体 | Kurtosis(cdk-erigon, cdk-node, zkevm-prover) | Kurtosis(op-node, op-geth, op-batcher, op-proposer) | conflux-rust(xjst 模式) |

`jsonrpc-proxy` 行的拓扑差异已在 4.4.2.1 给出结论，其根因论证延后至 4.4.3.2 完整展开。

`ydyl-gen-accounts` 不是可选辅助组件，而是与 L2 节点本体并列的核心组件。其定位体现在两方面。在实现层面：基于 Hardhat/TypeScript 实现，按 deterministic 派生规则生成账户——EVM 类链（CDK/OP）以 `chainID + index` 为 seed 派生私钥，XJST 以 `groupID + index` 为 seed 派生私钥——随后通过 L2 交易完成批量充值。在部署流程层面：每条链的 step9 调用一次，为该链产出 N 个可用测试账户。在验收目标层面：100 条链汇总后承担"全网累计创建 ≥10 亿用户账户"的规模化验收指标（见 4.4.1 研究目标 3），正是这一指标使 `ydyl-gen-accounts` 从辅助工具升格为核心组件。具体实现与单链账户量分配在 4.4.3 各栈小节中分别展开。

### 4.4.3 三类侧链流水线的服务与流程

#### 4.4.3.1 公共骨架：12 步流水线五阶段抽象

CDK/OP/XJST 三条流水线共享同一组 step 编号（XJST 因 L1 合约部署引入语义重排，见 4.4.3.4），其核心骨架可抽象为五个阶段。每个阶段的共性目的对所有栈一致，差异集中在阶段 ②③④ 的栈特定实现。

| 阶段 | 步骤 | 共性目的 |
|------|------|----------|
| ① 身份与资金准备 | step1 初始化密钥、step2 L1 充值 | 部署者 / claim-service / L2 部署者三类账户从 L1 vault 充值 |
| ② L1 接入层 | step3 启 jsonrpc-proxy（XJST：部 L1 合约） | 解决 4.2 节 L1 兼容性 / 部署本链 L1 合约 |
| ③ L2 部署 | step4 拉起 L2 节点、step5 L2 充值 | 链本体与 L2 测试钱包就绪 |
| ④ 跨链中继与桥注册 | step6 生成 .env、step7 部 Counter + 注册 bridge、step8 启 claim-service | 把单链转化为可互操作侧链 |
| ⑤ 元数据与可观测 | step9 批量账户、step10 元数据归档、step11 console-service、step12 PM2 巡检 | 为下游压测与控制平面查询提供契约 |

阶段之间的依赖关系决定了流水线的执行必须严格顺序：阶段 ① 为 ② 提供资金和身份，② 为 ③ 建立 L1 接入通道，③ 为 ④ 提供可操作的 L2 环境，④ 完成侧链的互操作转化，⑤ 的元数据产出依赖于前四阶段全部完成。三条流水线的差异主要落在阶段 ②③④——下面分别展开。

#### 4.4.3.2 OP Rollup 流水线

OP Stack 是本项目中与 4.2 节兼容性适配工作耦合最深的一栈——jsonrpc-proxy 的 block hash 修正直接塑造了 OP 链的部署拓扑。本节在三栈流水线中展开最详细，为 4.4.3.3（CDK）的对照讨论提供论述基础。

**部署的服务清单。** 每条 OP 链的部署涉及以下四类服务：

1. **全栈共享 `jsonrpc-proxy`**：所有 OP 链（2 条）共享同一个代理实例，由控制平面在独立位置部署一次，所有 OP 链的 `L1_RPC_URL` 均指向该实例。共享的根因在下文独立论证段展开。
2. **Kurtosis OP Stack**（每链一套）：通过 `optimism-package` 调用 Kurtosis 启动四进程——`op-node`（derivation + sequencer 协调）、`op-geth`（执行客户端）、`op-batcher`（calldata 批次提交至 L1）、`op-proposer`（output root 提交至 L1）。
3. **`op-claim-service`**（PM2，每链一套）：监听 L2 出口消息事件，构造 output root + Merkle proof 并在 L1 的 `OptimismPortal2` 合约上完成中继。
4. **`ydyl-gen-accounts`**（每链一次性运行）：在 step9 按 `l2type=0`（EVM 链）+ chainID 派生路径批量生成测试账户并完成 L2 充值。

**关键步骤深入。**

*step3 共享 proxy 接入。* step3 的语义不是"每条 OP 链各起一个 proxy"，而是"确保共享 proxy 已就绪并把其地址注入后续步骤的环境变量"。共享的根因来自 4.2 节 C1 的 hash 修正机制：jsonrpc-proxy 对 OP Stack 做 block hash 修正时，它从某个起始区块开始建立"Conflux 原生 hash → 以太坊标准 hash"的映射表。所有 OP 链的 op-node 在 derivation 管线中引用 L1 区块时，必须查询同一张映射表——若每条 OP 链各起独立 proxy，不同 proxy 的起始区块不同，同一 L1 区块在不同 proxy 上的修正后的 hash 可能分歧。在跨 OP 链消息验证场景中，两条 OP 链引用同一个 L1 区块却得到不同 hash，L1 引用的跨链一致性基础失效——这正是 4.4.2.1 中"OP 共享、CDK 独立"的拓扑差异的技术根因。

*step4 `step4_deploy_kurtosis_op`。* 通过 Kurtosis 的 `optimism-package` 拉起 OP Stack 四进程。两个关键配置参数由 4.2 节的适配工作决定：出块时间（`seconds_per_slot`）设为 1s——与 Conflux L1 约 1 秒的出块周期对齐，配合 4.2.4-B 的 L1 origin 追赶机制，将 4.4.1 C3 描述的 L2 head 滞后累积速率压缩到最小；batcher 强制 calldata 模式——规避 Conflux eSpace 未实现 EIP-4844 导致的 blob 交易不可用（4.2 节 C4）。

*step6 → step7 → step8 跨链链路。* 这三个步骤构成一条严格的依赖链。step6 从 `op-work/output/` 读取 L2 部署 key、claim-service EOA 和 bridge 注册所需参数，渲染出 `op-claim-service/.env` 与 `.env.counter-bridge-register` 两份配置文件。step7 调用 `zk-claim-service/scripts/i_deployCounterAndRegisterBridge.js`（命名带 zk 是历史原因，三栈共用），在 L2 上部署 Counter 合约并在 L1 的 `BRIDGE_HUB` 上注册 OP 标准桥——步骤细节见 4.4.3.5。step8 通过 PM2 启动 `op-claim-service`，进入"事件监听 → output root + Merkle proof 构造 → L1 `finalizeWithdrawalTransaction` 中继"的持续循环。

**OP 共享 proxy 与 CDK 独立 proxy 的根本差异。** 这一差异是 4.4 章最关键的架构决策之一，其根因需要三段分别论证。

OP 的 block hash 修正是有状态的。jsonrpc-proxy 对 OP Stack 的 hash 修正要求 proxy 从某个起始区块开始，维护"Conflux 原生 hash → 以太坊标准 hash"的映射关系。这一映射表的正确性条件不是每条链各自满足，而是所有 OP 链对同一个 L1 区块得到同一个修正后的 hash。两个独立 proxy 实例即便实现完全相同，只要起始区块不同，它们各自建立的映射表对历史区块的修正结果就可能出现分叉——因为 RLP 重算依赖完整的 15 字段区块头，而不同起始区块时刻获取的同一区块数据可能存在字段差异。这种分叉一旦发生，两条 OP 链在跨链消息验证中引用同一个 L1 区块却得到不同的 hash，跨链消息的 L1 锚定失去一致性基础。因此状态一致性要求强制所有 OP 链共享同一个 proxy 实例。

CDK 不在代理层做 hash 修正。4.2 节论证了 CDK 的 block hash 冲突由 `sync/evmdownloader.go` 的内核侧改造解决——在进入校验逻辑之前，直接用事件日志中的 `blockHash` 覆盖本地重算值，将比较双方统一为同一来源。代理层仅承担 RPC 转发与高 gasLimit 下的 gas price 补偿（详见 4.2.3.5），两者均为无状态操作。无状态意味着代理实例之间无须协调状态，每链独立部署一个 proxy 反而避免了单点瓶颈——一条 CDK 链的 proxy 失效不影响其他 CDK 链。

两种拓扑对部署架构的影响是直接的。OP 的共享 jsonrpc-proxy 是 100 链拓扑中除 `L1_BRIDGE_HUB_CONTRACT` 之外的第二个全局单点：该 proxy 实例失效将同时影响全部 2 条 OP 链的 derivation，使它们无法获取 L1 数据。CDK 则没有此类全局点，proxy 与节点 1:1 同生命周期。XJST 不需要 proxy——其 RPC 直连 L1 节点，无 hash 适配需求——三个栈在代理层的拓扑复杂度形成梯度：OP（全局共享，有状态）> CDK（每链独立，无状态）> XJST（无代理层）。

**OP 链 step9 的账户生成。** 每条 OP 链在 step9 调用 `ydyl-gen-accounts`。以 `l2type=0`（EVM 链类型标识）+ 该链的 chainID 作为 deterministic 派生 seed，批量生成私钥并通过 L2 交易完成充值。单链账户量是 10 亿总验收指标按 100 链摊分的份额，每条链承担约 1000 万账户的生成与充值。账户派生路径的 deterministic 特性保证相同 seed 可复现相同账户集，支持部署复现与审计验证。

#### 4.4.3.3 CDK Rollup 流水线

CDK 流水线与 OP 流水线共享 Kurtosis 部署框架和五阶段骨架，但在代理拓扑、跨链证明形态和资源消耗三个维度上存在关键差异。本节精简展开，仅给出差异点——与 OP 同型异构的部分用"同 4.4.3.2"一句带过，不再重复论证。

**部署的服务清单。** 每条 CDK 链所在 EC2 上部署以下四类服务：

1. **`jsonrpc-proxy`**（本链独立一实例，PM2 进程）：承担 RPC 转发与高 gasLimit 下的 gas price 补偿。CDK 不在代理层做 hash 修正（根因已在 4.4.3.2 论证段第二段完整给出），代理为无状态服务，每链独立部署避免单点瓶颈。
2. **Kurtosis CDK 容器栈**（每链一套）：`cdk-erigon`（执行节点）、`cdk-node`（sequencer + aggregator）、`zkevm-prover`（ZK proof 生成），通过 `kurtosis-cdk/external-l1.yml` 配置将 `settlement_backend` 设为 `l1`，aggregator 直接将 ZK proof 提交至 L1 上的 `PolygonRollupManager` 合约，绕过 Agglayer 服务（4.2.5-A 已论证）。
3. **`zk-claim-service`**（PM2，每链一套）：监听 L2 跨链出口事件，向 zkevm-prover 拉取 ZK proof，通过 L1 的 `PolygonRollupManager.verifyBatchesTrustedAggregator` 完成证明提交与中继。
4. **`ydyl-gen-accounts`**（每链一次性运行）：在 step9 以 `l2type=1`（CDK 链类型标识）+ chainID 派生路径生成测试账户。

**关键步骤要点。** step3 在本机拉起本链专属的 jsonrpc-proxy，将 `L1_RPC_URL` 改写为代理地址后注入后续步骤——与 OP 的"确保共享 proxy 已就绪"语义不同，CDK 是"每链各起一个"。step4 通过 `kurtosis run` 调用 `cdk-work/scripts/deploy.sh`，与 OP 共享 Kurtosis 编排框架但使用不同的 package（`kurtosis-cdk` 而非 `optimism-package`）。step6 → step7 → step8 链路的形态与 OP 同型异构：step6 渲染 `zk-claim-service/.env` 与 `.env.counter-bridge-register`，step7 调用同一个 Counter 部署与 bridge 注册脚本，step8 通过 PM2 启动 `zk-claim-service` 进入"事件监听 → ZK proof 拉取 → L1 提 claim"循环。与 OP 的差异仅在中继服务的证明形态——CDK 走 ZK proof，OP 走 Merkle proof。

**与 OP 的关键差异。** 三条差异集中在以下维度，不再单独论证：

1. **代理拓扑**：CDK 每链独立一个 proxy（已在 4.4.3.2 论证段第二段给出完整根因），无全局单点风险，proxy 与 CDK 节点 1:1 同生命周期。
2. **跨链证明形态**：CDK 走 ZK proof——`zk-claim-service` 通过 hardhat + ZK prover RPC 获取证明数据并在 L1 的 `PolygonRollupManager` 上提交；OP 走 Merkle proof——`op-claim-service` 通过纯 JS 计算 output root 与 Merkle proof，在 `OptimismPortal2` 上中继。两种证明形态的实现复杂度与 L1 gas 成本不同，CDK 单次提 claim 的 L1 成本高于 OP 的 Merkle proof 提交。
3. **资源消耗**：CDK 单节点在三栈中最重——多容器（cdk-erigon + cdk-node + zkevm-prover）叠加 ZK proof 生成的计算开销，要求更高的 EC2 机型配置。这一差异直接驱动 4.4.4 节 `ydyl-deploy-client` 的 `services[]` 按栈差异化配置机型与 `remoteCmd`。
