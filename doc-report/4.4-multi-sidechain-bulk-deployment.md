## 4.4 多类型侧链规模化部署

### 4.4.1 问题分析与研究目标

在异构 L2 栈共存的环境下，如何以可恢复、可观测、对后续跨链压测友好的方式，将 96 条 XJST、2 条 CDK（zk-rollup）、2 条 OP Stack（op-rollup）共 100 条侧链一次性部署到 AWS EC2 上，并保证产出的元数据足以驱动后续的跨链消息验证与 TPS 测试？

这一问题的挑战并非来自单一缺口，而是分布在异构栈差异、状态化部署长尾、网络拓扑约束和元数据契约四个维度。四类挑战各自独立——解决任意一类不能消除其余三类。

**C1 异构栈差异。** CDK 走 Kurtosis 编排的多容器栈（cdk-erigon / cdk-node / zkevm-prover），跨链中继依赖 ZK proof；OP Stack 同样走 Kurtosis 但容器组合不同（op-node / op-geth / op-batcher / op-proposer），跨链中继走 Merkle proof；XJST 走自研脚本部署 conflux-rust 的 xjst 模式节点，无独立中继服务，但需要在链内多节点间协调 L1 合约部署与地址分发。三套形态各自有独立的流水线脚本（`cdk_pipe.sh` / `op_pipe.sh` / `xjst_pipe.sh`）、独立的环境变量契约和独立的产物落点，不可能用单一脚本统一编排。

**C2 多步状态化部署。** 每条链 12 步流水线覆盖从密钥生成到 PM2 巡检的完整链路。单条 CDK 链的部署含 Kurtosis 多容器启动、L1 合约部署、claim-service 拉起等长尾步骤，单步失败不能从零重启——流水线必须支持断点续跑，且续跑时已完成的步骤不能重复执行，否则可能覆盖已有合约部署产物或重置 claim-service 状态。

**C3 拓扑约束。** 节点到 L1 RPC 的网络延迟一旦升高，OP/CDK 的 L2 derivation 拉取 L1 数据的速度会持续慢于 L1 出块速度，L2 所引用的 L1 head 与真实 L1 head 之间的滞后量随时间单调增长。这一滞后是累积型而非瞬时型——跨 region/AZ 拓扑下尤为明显，将"批量部署"从纯软件问题升级为部署拓扑约束。需要说明的是，4.2 节 C3 描述的是触达 `maxSequencerDrift` 硬阈值（1800 秒）后的 sequencer 瞬时停摆，而本节描述的是更早期、更普遍的 L2 head 持续滞后现象——两者是相邻而非同一问题。

**C4 元数据契约。** 100 条链各自产出 RPC 端点、bridge 合约地址、Counter 合约地址等元数据。若无统一格式与统一获取入口，下游的 `gen-cross-tx-config` 无法自动扫描所有链并生成跨链压测 jobs，部署产物与压测链路之间的人力手工衔接将成为规模化瓶颈。元数据契约的定义不仅关乎格式一致性，还要求每条链的 `ydyl-console-service` 健康运行以提供实时查询能力。

对应这四类挑战，本节的研究目标以四条量化指标定义：

1. 单条命令一次性发起 ≥100 条侧链批量部署；失败可定向重试不重建已成功节点
2. 部署完成后无人工整理即可生成 8 路并行跨链压测 jobs
3. 100 条链上累计创建 ≥10 亿用户账户——这一规模是侧链可承载真实业务负载的硬指标，由 `ydyl-gen-accounts` 在每条链的 step9 批量生成与充值
4. 96 XJST + 2 CDK + 2 OP，跨链 Counter 调用全链路可达
